# Javel日本小説構文解析器

原稿(テキストファイル)から構文木(AST)を返却するライブラリを作りたい。開発環境はBun。言語はTypeScript。原稿ファイル拡張子は`.jv`。UTF8。Unicode正規化形式NFC。改行コードLF。

### 表紙・裏表紙・目次

Javelファイルの冒頭には表紙がある。`＊＊＊`ブロックがあるとそれ以降は裏表紙。最初に出現する見出し`＃`までが表紙・裏表紙、それ以降が本文。目次は本文の見出しから作成する。

表紙は二つのブロックに分かれている。第一ブロックは題名のみ必須。それ以降の著者名、説明、NostrAddress or PubKey、LightningAddress、Nostr返信用発言IDは任意。第二ブロック以降は帯文として解釈する。帯文も任意。

見出しレベルは妥当性確認が必要。最初から見出しレベル２とかだと困る。前のレベルから２以上掘り下げたら困る。レベルは１〜６の範囲内まで。親方向に戻る時は制限なし。

章立ての数値は設定によって自動作成される。

### Nostrを使用して感想や指摘や投げ銭ができるようにする

1. nos2x-foxのようなSignerを自作する（複数アカウント管理可能。各種鍵、名前、アイコン画像などをIndexedDBに保存する）
2. ログインする

ブラウザの拡張機能で実装されたSignerを有効にしておく。`投稿する`ボタンを押下するとSignerのUIが起動する。ユーザを選んで決定する。ログインする。

```
[投稿する]
```

ログインすると以下の画面に変わる。

```
♡［面白かった！］［投稿］
[アイコン]→[アイコン]
　読者名　→　著者名　
```

この原稿に対する反応を受け取るNostr発言のIDを表紙に埋め込む。それに対して♡を与えるか返信できる。

♡を押下すると♥になる。返信はテキストエリアに入力した自由なテキストを返信できる。`投稿`ボタンを押下するとSignerのUIが出て、許可ボタンを押下したら投稿される。

読者のアイコン画像はSignerから取得する。著者のアイコン画像は原稿で設定したNostrAddressかPubKeyから取得する。

### 将来拡張するかもしれないもの

小説用であり原稿テキストで完結するのが望ましい。例外は設定や著者への感想・指摘・寄付のみ。なので基本的に拡張する気はないのだが。

#### リンク

```
＃https://...＃
＃テキスト｜https://...＃
＃１−２＃
＃テキストフラグメント＃
```

インライン要素。見出しと同じメタ文字を使っているが、これはインライン要素である。但しパーサにおいては区別できない場合がある。

```
＃見出しブロック

＃https://...＃のような段落ブロック。

区別がつかない！
```

`https://`が含まれているものは外部リンク。章の内部数値を指定しており`https://`が無いものは見出しの内部アンカーを示す。それ以外はテキストフラグメント。

この構文は追加しなくていいはずだ。小説は基本的に目次以外他の箇所に飛んだりしない。特に外部サイトへは基本的に飛ばない。せいぜい原稿への返信として各SNSサイトへ飛ばすくらいだ。それもNostrだけを使えば回避できる。よってこの構文は追加しなくていいはずだ。

#### 画像

```
！ここにALT属性値をセットする。
これ以降に画像URLやパスをセットする
/tmp/work/img/挿絵.png
https://www.some.com/img/123.png
段落が終わるまで好きなだけ。
上から順に検査し最初に実在が確認できたものを使う。
```

この構文は追加しなくていいはずだ。小説は基本的にテキストだけで構成すべきだ。アイキャッチや表紙のヘッダ画像だけは表紙で設定できるようにするが、本文内に置ける挿絵はないほうが良い。画像を使ってしまえば画像がメインコンテンツになってしまうからだ。

基本的に画像を入れることができるのは本文コンテンツが充実している場合のみだ。例えば表紙のヘッダ画像は本文が2万字以上のみとする。

挿絵にしても1章(2万字)あたりの見どころに一箇所のみというような制限を設けるべきだろう。

もし挿絵が豊富なら、テキストが適当でも不当に注目を浴びて小説としての評価を誤らせてしまう。クレバーにテキストだけの状態が最善だろう。

というわけで挿絵は追加しなくていい。なんならヘッダ画像すら微妙だが。まあ将来は特定のヘッダ画像を自動生成できるようにするかもしれないし。それだけは用意しておいていいかもしれない。

### 将来の文書構造解析

表紙・裏表紙・本文をどうにかして分割する必要がある。

1. 最初に発見した見出しレベル１の直前までを表紙・裏表紙とし、それ以降を本文と解釈する

原稿は以下3つに分割される。

* 表紙（必須）
* 裏表紙（任意）
* 本文（必須）

原稿は単なるテキストなので以下を確認する必要がある。

1. ファイル形式
    1. UTF8であること（文字セット）
    2. NFCであること（Unicode正規化形式）
    3. LFであること（改行コード）
2. 文書構造
    1. 表紙が存在すること
    2. 本文が存在すること

まずは原稿がこの構造であることを確認する。次に2〜3の要素である表紙・裏表紙・本文に原稿を大別する。そしてそれぞれに例のParserをかける。

表紙に関しては何段落目の何行目が何のデータであるか固定なので、その解析処理を実装する必要があるだろう。裏表紙に関してもNostrのUI挿入をする必要がある。

更には本文のパースが終わったら目次を作成する必要もある。全体的には以下の四項目とその順になる。

1. 表紙
2. 目次
3. 本文
4. 裏表紙

開発する時、どう責任分界点を割り出せばいいかが課題だ。

開発時の方針は以下。

* 事実に基づくこと
    * 全ての発言はその根拠を示せ
        * 公式サイトなどの完全なURLを示せ
* 論理に基づくこと
    * 全ての発言はその論理を文章で説明しろ
    * 誇張表現を含む形容は絶対に禁止する（`完全`,`100%`,`大正解`等）
* 単一責任の原則を死守する
    * 責任分界点を極限まで細分化しろ
* DRYに書く
    * 再利用可能部品へと細分化しろ

尚、Parser完了後には将来Rendererを作り、更に表示設定するStyler、それらを元に画面サイズごとにページ分割するPagerも開発する予定。そのベースとなる構文木返却が今回の守備範囲である。
