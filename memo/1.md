### 1. 基礎定義と規格準拠

#### 1.1. 入力仕様の定義
*   **文字符号化方式**: UTF-8
*   **正規化形式**: Unicode Normalization Form C (NFC)
    *   根拠: [Unicode Standard Annex #15](https://unicode.org/reports/tr15/) において、正規化はテキスト比較や処理の標準化に必須とされる。
*   **改行コード**: LF (`\n`)
*   **ファイル拡張子**: `.jv`

#### 1.2. 責任分界点の定義
解析処理を以下の4つの独立したフェーズに分割します。

1.  **Normalizer**: 入力文字列の正規化（NFC変換、改行コード統一）。
2.  **SectionSplitter**: 原稿を「表紙」「本文」「裏表紙」に分割。
3.  **Tokenizer**: 文字列を最小単位のブロック（Token）に分割。
4.  **Lexer**: Tokenを解析し、意味情報を持つ構文木（AST）へ変換。

### 3. 実装コンポーネントの細分化

#### 3.1. SectionSplitter (セクション分割)
原稿を構造的に分割する責務を持ちます。

*   **論理**: 
    1.  最初の「見出し1（`＃ `）」が出現するまでを「表紙・裏表紙領域」とする。
    2.  その領域内で「`＊＊＊`（主題区切り）」が存在する場合、それ以前を表紙、以降を裏表紙とする。
    3.  最初の見出し以降を本文とする。

#### 3.2. Tokenizer (字句解析器)
正規表現を用いず、1文字ずつ走査（スキャン）することで、エスケープ処理を確実に実行します。

*   **単一責任**: 文字列を「意味のある塊」の境界で切断することのみに集中する。
*   **DRYの適用**: フェンス（` ``` `等）の検出ロジックを共通化し、開始マーカーと終了マーカーの対を管理するスタック構造を採用する。

#### 3.3. Lexer (構文解析器)
Tokenの種類に応じて、具体的なAstNodeを生成します。

*   **HeadingLexer**: 見出しレベルの妥当性（1〜6、前レベルからの乖離）を検証。
    *   根拠: HTMLのアクセシビリティ指針（[W3C H42](https://www.w3.org/WAI/WCAG21/Techniques/html/H42)）に準じ、階層構造の整合性を維持する。
*   **InlineLexer**: 段落内のルビ、傍点、割注を解析。
    *   **割注の論理**: 全角を2、半角を1として長さを計算。パイプ（`｜`）による明示的分割がない場合、中央値で分割。
    *   根拠: [日本語組版処理の要件（JLReq） 3.4.8](https://www.w3.org/TR/jlreq/#subclause_3_4_8) における割注の配置規則を参照。

``````markdown
ご指摘いただいたメモリ効率、ストリーミング処理の必要性、およびトークン・ASTの厳密な定義に基づき、設計を再構築します。

### 1. ストリーミング解析とセクション判定の論理

10万字規模の原稿をフリーズさせずに処理するため、`parse()` は `Generator`（または `AsyncGenerator`）として実装し、トークン単位で `yield` します。

#### 1.1. セクション状態の遷移（State Machine）
「表紙・裏表紙・本文」の区別は、パーサ内部に状態（State）を持たせることで解決します。パーサは現在どのセクションを解析しているかのメタ情報を付与して `AstNode` を返却します。

*   **状態定義**: `FRONT`（表紙）, `BACK`（裏表紙）, `BODY`（本文）
*   **遷移論理**:
    1.  初期状態は `FRONT`。
    2.  `FRONT` 状態で `＊＊＊` (ThematicBreak) を検出したら `BACK` へ遷移。
    3.  最初の `＃ ` (Heading Level 1) を検出した瞬間、それ以降は強制的に `BODY` へ遷移。
*   **利点**: 消費側（レンダラー等）は、`yield` されたノードの `section` プロパティを見るだけで、それがどこに属するか即座に判断できます。

#### 1.2. 目次（ToC）と裏表紙の配置問題
*   **目次**: ストリーミングの性質上、パース完了前には目次は完成しません。そのため、目次は「パース中に見出しノードを別途収集する」という**二次的な責務**として分離します。
*   **配置**: 原稿上の順序（表紙→裏表紙→本文）と表示上の順序（表紙→目次→本文→裏表紙）が異なる問題は、パーサの責務ではなく、**レンダラー（Pager）の責務**とします。パーサはあくまで「原稿の出現順」に忠実にノードを流します。

---

### 2. トークン定義の厳格化

メモリ節約のため、文字列のコピー（`raw`）を保持せず、元の原稿文字列へのインデックスのみを保持します。

```typescript
export type TokenKind = 'fence' | 'block' | 'space' | 'inline';

export interface BaseToken {
    kind: TokenKind;
    start: number; // 原稿内の開始オフセット
    end: number;   // 原稿内の終了オフセット
}

export interface FenceToken extends BaseToken {
    kind: 'fence';
    marker: string; // メタ文字（"`", "＠", "＝" 等）
    count: number;  // メタ文字の数（3以上）
    // コンテンツの開始・終了位置（フェンス記号自体を含まない範囲）
    contentStart: number;
    contentEnd: number;
}

export interface SpaceToken extends BaseToken {
    kind: 'space';
    count: number; // 改行の数（3以上）
}
```

*   **フェンスの論理**: 
    *   行頭から始まること。
    *   開始と終了の `marker` および `count` が一致すること。
    *   ネストは「親より `count` が少ないもの」のみ許容するスタック管理を行う。
*   **Spaceの論理**: 
    *   3連続以上の改行を一つの `SpaceToken` とする。
    *   これは「意図的な空白」として保持し、後の工程で削除・維持を選択可能にする。

### 3. ASTノード構造

`token` オブジェクトをそのまま保持し、原稿の完全復元を可能にします。

```typescript
export interface AstNode {
    /** 元となったToken。start/endにより原稿から1文字の狂いなく復元可能 */
    token: BaseToken;
    /** 解析された意味データ（見出しレベル、ルビの親文字と読み等） */
    data: Record<string, any>;
    /** 表記の揺れやエスケープ時の全角スペース除去情報など、復元に必要な情報 */
    notation: Record<string, any>;
    /** セクション情報: 'front' | 'back' | 'body' */
    section: string;
    /** 子ノード（段落内のインライン要素やフェンス内の要素など） */

}
```

### 4. 実装構造（単一責任の原則に基づく）

#### 4.1. Javelクラス（オーケストレーター）
`parse` メソッドは、内部の状態を管理しながらノードを `yield` します。

```typescript
export class Javel {
    private state: 'front' | 'back' | 'body' = 'front';
    private hasHeading1Occurred = false;

    async *parse(manuscript: string): AsyncGenerator<AstNode> {
        // Tokenizerはイテレータとして実装し、一気にメモリに載せない
        const tokens = this.tokenizer.tokenize(manuscript);

        for (const token of tokens) {
            const node = this.lexer.lex(token);
            
            // セクション状態の更新論理
            this.updateState(node);
            node.section = this.state;

            yield node;

            // 100msごとの待機などの制御は、このGeneratorを回すループ側（UIスレッド側）で行う
            // 必要に応じてここに協調的マルチタスクのyieldを挟むことも可能
        }
    }

    private updateState(node: AstNode) {
        if (!this.hasHeading1Occurred) {
            if (node.type === 'heading' && node.data.level === 1) {
                this.state = 'body';
                this.hasHeading1Occurred = true;
            } else if (this.state === 'front' && node.type === 'thematic_break') {
                this.state = 'back';
            }
        }
    }
}
```

### 5. 論理的根拠のまとめ

1.  **メモリ効率**: `substring()` による部分文字列の生成を避け、`start/end` インデックスで管理することで、巨大な原稿でもメモリ消費を最小限に抑えます。
    *   根拠: [V8エンジンの文字列スライス最適化](https://v8.dev/blog/hash-code#strings)（スライスは元の文字列への参照を持つ場合があるが、明示的なインデックス管理の方が確実）。
2.  **非フリーズ**: `AsyncGenerator` を採用することで、呼び出し側が `for await` ループ内で `setTimeout` 等による割り込みを入れる余地を作ります。
3.  **完全復元性**: `AstNode` が `Token`（位置情報）を保持し続けることで、エディタ上でのハイライトや、特定のHTML要素から原稿の該当箇所を特定するリバースマッピングを可能にします。
4.  **単一責任**: パーサは「原稿の構造化」に専念し、表示順序の入れ替えや目次の構築は、パーサから流れてくるイベントを受け取る「コレクター」や「レンダラー」の責務とします。
``````
